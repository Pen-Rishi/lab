CONNECTSECURE SCAN PIPELINE — WHERE TO ADD EACH CHECK
=====================================================


STAGE 1: PORT SCAN -> SERVER FINGERPRINTING
============================================

When: After nmap -sV returns service detection, BEFORE mapping vulns to services

Add this check:

  if server_header == "nginx" or "cloudflare" or "awselb":
      DO NOT report Apache Struts findings
      DO NOT report IIS-specific findings
      DO NOT report Apache-specific findings

  if server_header == "Apache":
      OK to report Apache findings

Why: CS currently detects "Apache Struts" on an nginx server. This kills
    3 false findings immediately.


STAGE 2: URL DISCOVERY -> FP FILTER
====================================

When: After fetching a URL, BEFORE running any injection tests on it

Add this check:

  response = GET /some-page

  if response.status_code is 301, 302, 307, or 308:
      SKIP - page does not exist, it's a redirect
      DO NOT report ANY findings on this path

  if response.body contains any of:
      "404 Not Found"
      "Page not found"  
      "Route inconnue"
      "does not exist"
      "Error 404"
      "No such page"
      "Sorry, the page"
  then:
      SKIP - server returned 200 but it's an error page
      DO NOT report ANY findings on this path

Why: This single check kills ALL webshell FPs, clickjacking FPs, 
    and any path-based findings on non-existent pages. ~15 FPs gone.


STAGE 3: INJECTION TESTING (XSS, SSTI, NoSQL, Command Injection)
=================================================================

When: After sending a payload, BEFORE reporting it as a finding

CURRENT (wrong):
    if payload_text in response.body → report as vulnerability

FIX (correct):
    if payload_text in response.body:
        
        CHECK WHERE it appears:
        
        if payload only appears in:
            - <a href="/?param=PAYLOAD">
            - page text that just echoes the URL
            - error message showing the URL
        then:
            report as "Reflected Parameter" (INFO)
            DO NOT report as XSS, SSTI, NoSQL, etc.
        
        if payload appears as raw HTML/JS outside URL context:
            report as actual vulnerability

For SSTI specifically ({{7*7}}):

    if response.body contains "49" AND "{{7*7}}" is still visible in response:
        SKIP - "49" is from URL reflection, not template evaluation
        // If server evaluated the template, {{7*7}} would be GONE,
        // replaced by 49. Both appearing = FP.
    
    if response.body contains "49" AND "{{7*7}}" is NOT in response:
        report as SSTI - template was actually evaluated

Why: Fixes all SSTI, NoSQL, Prototype Pollution, and generic injection FPs.
    ~25 FPs gone.


HOST HEADER INJECTION TEST
---------------------------

When: After server fingerprinting, BEFORE reporting "no host header injection"

Add this check:

    known_falsy_hosts = [
        "127.0.0.1",
        "localhost",
        "0.0.0.0",
        "evil.com",
        "attacker.com",
        "burpcollaborator.net"
    ]
    
    for test_host in known_falsy_hosts:
        response = GET / with Host: test_host
        
        CHECK WHERE test_host appears in response:
        
        if test_host appears in response.body inside:
            - <a href="http://test_host/...
            - <link rel="canonical" href="http://test_host/...
            - <form action="http://test_host/...
            - <img src="http://test_host/...
            - Location: http://test_host/... (redirect header)
            - <base href="http://test_host/...
        then:
            report as "Host Header Injection — Server Reflects Host Value in Output" (HIGH)
            include:
                - the test host used
                - the exact HTML/Location context where it appeared
                - a note: "This enables cache poisoning attacks. Attacker sets Host header to
                  attacker-controlled domain, cached page serves malicious URLs, user clicks
                  and reaches attacker site. Chain to XSS if cached page loads attacker JS."
        
        if NONE of the test hosts appear in any URL context:
            report as "No Host Header Injection Found" (PASS)

Why: Host header injection is the highest-value finding on the 159.65.95.70 target
    and ALL THREE scanners (CS, Tenable, Qualys) missed it. Every asset URL on the
    target reflects the Host header verbatim, enabling cache poison → XSS chains.
    CS can be the first scanner to catch this systematically.


STAGE 4: STORED XSS DETECTION
==============================

When: After POSTing a payload and checking the response

CURRENT (wrong):
    POST payload to page
    Check SAME response for marker → report stored XSS

FIX (correct):
    Step A: POST payload to page (body has marker? ignore this)
    Step B: GET the SAME page WITHOUT sending any payload
    
    if marker in Step B response AND marker was NOT sent in Step B:
        report as Stored XSS - payload persists across requests
    else:
        SKIP - marker only appeared because we just POSTed it
        This is reflected in the same request, not stored

Why: Fixes all 40+ stored XSS FPs. Payload appears in POST response
    because the server echoes it back, not because it's stored.


STAGE 5: SKIMMER / MALWARE DETECTION
=====================================

When: Analyzing external JavaScript files

CURRENT (wrong):
    if "fromCharCode" in JS file → report skimmer
    if "google-analytic" in page → report magecart

FIX (correct):

    First, check if domain is known-safe:
    
    KNOWN_SAFE_CDNS = [
        "cdnjs.cloudflare.com",
        "js.stripe.com", 
        "connect.facebook.net",
        "js.sentry-cdn.com",
        "static.klaviyo.com",
        "checkout.razorpay.com",
        "unpkg.com",
        "cdn.faceit.com",
        "html5shim.googlecode.com",
        "cdn.jsdelivr.net",
        "ajax.googleapis.com",
        "cdn.optimizely.com"
    ]
    
    if script_domain in KNOWN_SAFE_CDNS:
        SKIP - do not report skimmer on trusted CDNs
    
    Second, check for real skimmer signatures:
    
    REAL_SKIMMER_PATTERNS = [
        "atob(",                          // base64 decode (common in skimmers)
        "document.getElementById",        // form field collection
        ".value",                         // form value extraction
        "exfiltration", "exfil",          // exfiltration
        "sendBeacon",                     // beacon exfil
        "https://evil",                   // external exfil URL
        "steal", "skimmer"
    ]
    
    if only "fromCharCode" matched:
        SKIP - this is a standard JS function, 100% of JS uses it
    
    if real skimmer patterns match AND script is from unknown domain:
        report as potential skimmer

Why: Fixes all skimmer FPs. fromCharCode appears in every JS file.
    Stripe, Facebook, Sentry are not running skimmers.


STAGE 6: DEDUPLICATION
=======================

When: After all scanning, BEFORE generating the final report

Add these grouping rules:

    Rule 1: Same script on port 80 and 443
        if same_script_url missing SRI on port 80 AND 443:
            merge into ONE finding: "script.js missing SRI (ports 80, 443)"
    
    Rule 2: Same CDN domain on both ports
        if same_cdn flagged as "unknown domain" on port 80 AND 443:
            merge into ONE finding
    
    Rule 3: TLS version + TLS version PCI DSS
        if "TLS 1.0" and "TLS 1.0 (PCI DSS)" both reported:
            merge into ONE finding with PCI DSS note
    
    Rule 4: Same vulnerability on multiple event pages
        if same vuln on /event/amsterdam2026, /event/sav-vampire-ball-2026, etc.:
            merge into ONE finding: "Vulnerability on /event/* pages (X endpoints affected)"

Why: Fixes 25+ duplicate findings. Makes report 60% shorter.
    Client doesn't need to see the same issue 12 times.
